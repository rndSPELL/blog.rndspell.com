Компонентная система сущностей

В процессе пробы пера в написании игр я начал задаваться вопросом, как же организовать структуру классов игры, как правильно выстроить иерархию объектов и архитектуру игры в целом. Конечно, я руководствовался примерами из учебной литературы, форумов, реальными opensource играми, но всегда получал на выходе, как мне казалось, неповоротливых монстров. Скорее всего это от неопытности. Но постепенно я понял, что простое наследование классов в постоении иерархии объектов в игре никогда не приведет меня к желаемому результату. Я всегда имел примерно следующее:

public abstract class BaseObject(){
    String name;
}

public abstract class CollidableObject extends BaseObject(){
    Rectangle bounds;
}

public class Bullet extends CollidableObject(){
    float velocity;
}

public class Tree extends BaseObject(){
    float height;
}

Вроде бы ничего страшного пока нет, и вполне все логично. Но что если далее ввести класс Creature, наследники которого могут как участвовать в расчете коллизий, так и нет. Т.е. в этом случае надо вводить два класса? (В java нет множественного наследования)

public abstract class Creature extends BaseObject(){}

public abstract class CollidableCreature extends CollidableObject(){}

Или использовать, допустим, интерфейс Collidable и реализовывать его для объектов, участвующих в коллизиях?

Ну в общем так и пишется большинство игр. Иерархия классов продумывается заранее, используются наработки, опыт, чтобы это все не погрязло в дебрях наследований, и, поверьте, это все работает. Возможно я страдаю от заблуждений молодых программистов, которые хотят иметь красивый, масштабируемый код, порядок в структуре проекта и наглядность без лишних комментариев и многостраничной документации, при этом хотят еще и иметь возможность дописать в будущем функционал (который еще не придуман) к игре без лишних телодвижений, предусмотреть всевозможные методы заранее и т.д. и т.п. Да это все имеется и у меня, хотя я старательно убиваю в себе перфекциониста, но все равно хочу быть удовлетворенным итоговым кодом. Классическое наследование объектов от общего к частному в игре меня не устраивает. Постепенно пришло осознание, что необходима некая фабрика игровых объектов из блоков, отвечающих за определеный аспект игры: такие как отображение на экране, коллизии, игровые характеристики. Конечно, я был уверен, что все уже придумано до меня, поэтому быстро нашел информацию по Компонентной Системе Сущностей и понял - это то, что я искал!

Основная идея системы сущностей - это сущность, которая является контейнером компонентов, определяющих состояния сущности (положение, размер, вес, скорость, используемая текстура и т.д.), обновление которых происходит в "системах". При этом получаем некую модульность и независимость компонент и систем друг от друга, т.е. можно управлять одним аспектом сущностей, независимо от другого (менять положение в системе "СистемаПередвиженияОбъектов" при этом это не зависит от работы системы "СистемаОтображенияОбъекта"). Данные для определения состояний компонентов могут хранится в любом виде (база данных, xml, и т.д). Различные игровые объекты формируются путем комбинации компонентов, при этом объекты можно формировать на лету.

Вот неплохая статья (оригинал) для ознакомления с основными принципами архитектуры системы сущностей. Автор оригинальной статьи написал собственный фреймворк Ash.

У badlogic имеется фреймворк компонентной системы сущностей (КСС), написанный на Java, называется он Ashley и черпает идеи из Ash (Название как бы намекает). Очень простой, понятный код позволяет понять принцип работы КСС в реальном приложении. Очень рекомендую ознакомиться с исходным кодом. Вот им я и воспользуюсь.

Используем систему сущностей в проекте

Подключение к текущему проекту элементарное, в нашем случае просто прописывается зависимость в build.gradle:

project(":core") {
    ...

    dependencies {
        ...
        compile "com.badlogicgames.ashley:ashley:1.0.1"
    }
}

и перезапускается проект. Теперь можно использовать фреймворк.

После ознакомления с емкой документацией в виде readme в репозитории фреймворка Ashley можно приступать к написанию компонент для будущих игровых объектов.

public class PositionComponent extends Component implements Json.Serializable {

    private Vector2 position;

    public PositionComponent(){
        position = new Vector2();
    }

    public PositionComponent(Vector2 position){
        this.position = position;
    }

    @Override
    public void write(Json json) {}

    @Override
    public void read(Json json, JsonValue jsonData) {
        json.readField(this, "position", jsonData);
    }

}

Компонента PositionComponent наследует класс Component фреймворка, имеет всего лишь одно поле position, пару конструкторов и реализацию методов интерфейса Json.Serializable для возможности загрузки данных из файлов Json (об этом поговорим в будущем), геттеры и сеттеры опустим. Добавим еще пару необходимых компонентов.

Компонент, отвечающий за движение MovementComponent.java:

public class MovementComponent extends Component implements Json.Serializable {

    private Vector2 velocity = new Vector2();
    private Vector2 acceleration = new Vector2();
    private float accelerationValue;
    private float maxVelocity;

    public MovementComponent(float accelerationValue, float maxVelocity){
        this.accelerationValue = accelerationValue;
        this.maxVelocity= maxVelocity;
    }

    @Override
    public void write(Json json) {}

    @Override
    public void read(Json json, JsonValue jsonData) {
        json.readField(this, "accelerationValue", jsonData);
        json.readField(this, "maxVelocity", jsonData);
    }

}

вектор velocity определяет текущие скорость и направление движения объекта, maxVelocity - это максимальная скорость объекта, acceleration вектор ускорения с значением равным accelerationValue. Т.е. двигать объект будем посредством ускорения.

За отображение на экране отвечает RenderComponent.java:

public class RenderComponent extends Component implements Json.Serializable {

    private TextureRegion frame = GameAssetManager.getInstance().get("textures/textures.atlas", TextureAtlas.class).findRegion("null");
    private Vector2 frameOffset = new Vector2();

    public RenderComponent(){}

    public RenderComponent(Vector2 offset){
        this.frameOffset = offset;
    }
    public RenderComponent(TextureRegion texture, Vector2 offset){
        this.frame = texture;
        this.frameOffset = offset;
    }

    @Override
    public void write(Json json) {

    }

    @Override
    public void read(Json json, JsonValue jsonData) {
        json.readField(this, "frameOffset", jsonData);
    }

}

Поле frame - текущая текстура для отображения (для анимированных объектов она будет обновляться), сразу задаю текстуру "по умолчанию". Поле frameOffset это смещение текстуры относительно положения объекта.

И да, текстуру теперь мы получаем из атласа TextureAtlas, пожалуй остановлюсь на этом поподробнее.

Текстуры для игры, если это маленькая игра с малым числом объектов на экране, без анимации, можно хранить в виде отдельных изображений, которые загружаются в память и отображается по мере необходимости. Но когда таких текстур будет много, появятся многокадровые анимации, загружать и использовать в процессе рендеринга такие текстуры довольно накладно. Правильно хранить текстуры в составе больших изображений, которые один раз загружаются и видеокарта просто берет необходимый участок уже из загруженного изображения. Работать с атласами во фреймворке libGDX нужно посредством класса TextureAtlas. Сформировать атлас можно разными способами, самый удобный - воспользоваться утилитой libgdx-texturepacker.

А получаем текстуру из менеджера следующим образом:

где метод findRegion в качестве аргумента принимает имя исходного файла текстуры без расширения.

Итак, продолжим. Для изменения состояний компонетов, т.е. для изменения положения, скорости/ускорения и отрисовки нам нужны системы. Вот они:

RenderSystem.java

public class RenderSystem extends EntitySystem {

    private ImmutableIntMap<Entity> entities;

    private OrthographicCamera camera;
    private SpriteBatch batch;

    public RenderSystem(OrthographicCamera camera){
        this.camera = camera;
        this.batch = new SpriteBatch();
    }

    @Override
    public void addedToEngine(Engine engine) {
        entities = engine.getEntitiesFor(Family.getFamilyFor(PositionComponent.class, RenderComponent.class));
    }

    @Override
    public void removedFromEngine(Engine engine) {
        super.removedFromEngine(engine);
    }

    @Override
    public void update(float deltaTime) {

        PositionComponent position;
        RenderComponent render;

        Gdx.gl.glClearColor(0, 0, 0, 1);
        Gdx.gl.glClear(GL20.GL_COLOR_BUFFER_BIT);
        batch.setProjectionMatrix(camera.combined);
        camera.update();

        batch.begin();

        for(Entity entity: entities.values()){
            position = entity.getComponent(PositionComponent.class);
            render = entity.getComponent(RenderComponent.class);
            batch.draw(render.getFrame(), position.getPosition().x, position.getPosition().y);
        }

        batch.end();
    }

}

MovementSystem.java

public class MovementSystem extends IteratingSystem {

    private Random random = new Random();

    public MovementSystem() {
        super(Family.getFamilyFor(PositionComponent.class, MovementComponent.class));
    }

    @Override
    public void processEntity(Entity entity, float deltaTime) {
        PositionComponent position = entity.getComponent(PositionComponent.class);
        MovementComponent movement = entity.getComponent(MovementComponent.class);

        // для проверки работы системы зададим движение в случайном направлении
        if(random.nextInt(10) < 1){
            movement.getAcceleration().set(movement.getAccelerationValue(), 0).rotate(random.nextInt(360));
        }

        movement.getVelocity().add(movement.getAcceleration().cpy().scl(deltaTime));
        if(movement.getVelocity().len() > movement.getMaxVelocity()){
            movement.getVelocity().limit(movement.getMaxVelocity());
        }
        position.getPosition().add(movement.getVelocity().cpy().scl(deltaTime));
    }
}

Для создания систем нам предоставляют два класса, от которых можно их наследовать: IteratingSystem и EntitySystem. Первое используется, если необходимо просто в цикле обновлять каждую сущность из списка, система MovementSystem как раз подходит под этот случай. Если же при обновлении системы помимо итерации по списку объектов нужно проделать еще какие-либо действия, как например в классе RenderSystem, тогда лучше воспользловаться классом EntitySystem

В системе рендеринга вы уже увидели некую камеру OrthographicCamera, это именно то, что будет представлять камеру в игре, ее можно двигать, приближать и удалять, крутить и т.д.

Список объектов, которые необходимо обрабатывать в системе, получаем отбором сущностей, имеющих требуемый системой набор компонентов:

entities = engine.getEntitiesFor(Family.getFamilyFor(PositionComponent.class, RenderComponent.class));

Такой отбор необходим для увеличения быстродействия, поскольку с ростом кол-ва объектов, обработка всех их в каждой системе приводила бы быстрому росту затрат вычислительной мощности. Далее устанавливаем для SpriteBatch проекционную матрицу камеры, т.е. делаем вид из камеры и обновляем ее.

batch.setProjectionMatrix(camera.combined);
camera.update();

И уже в цикле отображаем по очереди все сущности из списка:

batch.begin();

for(Entity entity: entities.values()){
    position = entity.getComponent(PositionComponent.class);
    //получаем компонент положения обекта в игровом мире
    render = entity.getComponent(RenderComponent.class);
    //получаем компонент, отвечающий за отображение объекта
    batch.draw(render.getFrame(), position.getPosition().x, position.getPosition().y);
    //отрисовываем, используя полученные выше компоненты
}

batch.end();

Теперь рассмотрим систему MovementSystem.

movement.getVelocity().add(movement.getAcceleration().cpy().scl(deltaTime));

в данной строчке происходит ускорение, т.е. вектор скорости складывается с вектором ускорения, умноженным на промежуток времени между двумя кадрами deltaTime. Метод cpy() возвращает копию вектора (мы же не хотим изменять вектор ускорения, скалярно умножая его scl() на время deltaTime с каждым циклом).

if(movement.getVelocity().len() > movement.getMaxVelocity()){
    movement.getVelocity().limit(movement.getMaxVelocity());
}

Ограничиваем величину вектора скорости максимальным значением maxVelocity.

position.getPosition().add(movement.getVelocity().cpy().scl(deltaTime));

И, наконец, изменяем положение объкта в игровом мире на величину скорости умноженным на время (никаких неожиданностей), опять же берем копию скорости.

Для проверки работы системы введем случайное изменение направления вектора ускорения в одном такте из десяти:

if(random.nextInt(10) < 1){
    movement.getAcceleration().set(movement.getAccelerationValue(), 0).rotate(random.nextInt(360));
}

Итак, у нас есть три компонента и две системы, с помощью которых можно создавать двигающиеся или статичные объекты, которые можно отрисовать на экране. Перейдем к экрану игры и создадим один такой объект. Опять обращаемся к документации к фреймворку Ashley, смотрим пример и повторяем в GameScreen.

public class GameScreen implements Screen {

    Engine engine;
    OrthographicCamera camera;

    public void render(float delta) {

        engine.update(delta);

    }

    @Override
    public void show() {

        camera = new OrthographicCamera(640, 480);

        engine = new Engine();

        engine.addSystem(new MovementSystem());
        engine.addSystem(new RenderSystem(camera));

        Entity entity = new Entity();
        entity.add(new PositionComponent());
        entity.add(new MovementComponent(8f, 32f));
        entity.add(new RenderComponent(new Vector2()));

        engine.addEntity(entity);
    }

    ...

    }
}

Разрешение камеры задаем 640х480, это и будет разрешение игры, т.е. в полноэкранном режиме мы получим растянутое изображение. Engine это основной класс фреймворка Ashley, в нем регистрируются сущности и системы, он так же имеет метод update(), который необходимо вызывать в главном цикле игры.
